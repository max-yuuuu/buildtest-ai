## 一、问题陈述

BuildTest AI 已具备文档解析、切分、向量入库等 ingestion 能力，但 query-time 读取流程尚未形成可稳定交付的产品链路。当前 Quick Chat MVP 使用模板拼接生成回答（无 LLM 调用），agent/data 模式返回 `MODE_NOT_IMPLEMENTED`。

**核心问题：** 用户上传了文档、建好了知识库，但无法真正"对话式"地从中获取知识。这使得整个平台停留在"数据管理工具"阶段，无法体现 RAG 的核心价值。

**目标：** 落地两条只读工作流：
1. **快速问答（Quick Chat）** — 检索 → 生成 → 引用 → 流式输出，单轮对话
2. **智能 Agent（Smart Agent）** — ReAct 循环，多步推理，工具调用，多轮对话

### 2.3 参考架构对比

**WeKnora（Go 版本）核心流程：**
```
查询理解 → 知识检索(向量+BM25+混合) → 重排序 → 答案生成(流式)
Agent: Think → Act(工具调用) → Observe → 循环/结束
```

**RAGAnything Agentic RAG：**
```
查询分析 → 多模态内容分析 → 检索增强(向量+知识图谱) → 推理与决策 → 响应合成
```

**我们的目标（Python + LangGraph）：**
```
Quick: normalize → retrieve → assemble → LLM generate(stream) → citations
Agent: init → build prompt → [Think → ToolCall → Observe]×N → final answer
```

## 三、前提条件（Premises）

1. **后端主导工作流，BFF 只做协议映射** — Python 侧用 LangGraph 编排，Next.js BFF 将领域事件映射为 AI SDK UI Message Stream
2. **Quick Chat 先行，Agent 跟进** — Quick 是最小可用链路，Agent 是扩展
3. **LLM Provider 可配置** — 通过已有的 `providers` 表和 `model_service` 获取 LLM 模型配置
4. **向量检索已有，不重新实现** — 复用 `NaiveV1RetrievalStrategy` + `VectorDbConnector`
5. **流式协议已对齐** — 后端 SSE 事件 → BFF 映射 → AI SDK 6 的链路已打通
6. **不做数据库持久化** — 本期不引入 chat_sessions/chat_messages 表
7. **单用户开发阶段** — 暂不考虑并发和多租户隔离的性能优化

## 四、方案对比

### 方案 A：最小可行（Minimal Viable）

**摘要：** 在现有 Quick Chat Workflow 基础上，替换模板生成为 LLM 调用，不引入 LangGraph。

**工作量：** S（~2 天）
**风险：** Low

**优点：**
- 改动最小，只改 `quick_chat_use_case.py` 中的生成节点
- 复用全部已有代码（路由、流式、前端）
- 快速验证端到端链路

**缺点：**
- 无状态机，后续 Agent 模式需要重构
- 无法支持多步推理、工具调用循环
- Query Rewrite 质量无法保证

**复用：** 全部现有 chat 架构

### 方案 B：LangGraph 统一编排（Ideal Architecture）

**摘要：** 引入 LangGraph，用状态图编排 Quick 和 Agent 两种工作流。Quick 是线性图，Agent 是带循环的图。

**工作量：** M（~5 天）
**风险：** Med

**优点：**
- 统一的状态机编排，Quick/Agent 共享基础设施
- Agent 模式天然支持 ReAct 循环（Think → Act → Observe）
- 可观测性好（LangGraph 内置 trace）
- 与 WeKnora 架构对齐，后续可渐进扩展

**缺点：**
- 引入新依赖（langgraph, langchain-core）
- 需要重构现有 workflow 为 LangGraph 节点
- 学习曲线

**复用：** 检索策略、Tool Registry、流式协议、前端

### 方案 C：LangGraph + 对话历史 + Rerank

**摘要：** 在方案 B 基础上，增加对话历史管理和检索重排序。

**工作量：** L（~10 天）
**风险：** High

**优点：**
- 完整的产品能力
- 多轮对话体验好
- 检索质量高

**缺点：**
- 引入数据库变更（chat_sessions, chat_messages）
- 复杂度高，MVP 周期长
- 过度设计风险

**复用：** 方案 B 的全部 + 新增 DB 表

### 推荐

**选择方案 B** — LangGraph 统一编排。理由：
1. 方案 A 虽然快，但 Agent 模式必须重写，属于"还债型"代码
2. 方案 C 引入 DB 变更，与"本期不做持久化"的前提矛盾
3. 方案 B 在不引入 DB 变更的前提下，同时支持 Quick 和 Agent 两种模式
4. LangGraph 是 Python RAG/Agent 的事实标准，投入学习成本有长期回报

#### Agent Chat 图

```python
# State 定义
class AgentChatState(TypedDict):
    query: str                          # 用户查询
    knowledge_base_ids: list[str]       # 目标知识库
    messages: list[BaseMessage]         # 对话历史（LangChain 消息）
    tool_calls: list[ToolCallRecord]    # 工具调用记录
    citations: list[Citation]           # 引用映射
    answer: str                         # 最终回答
    iteration: int                      # 当前迭代次数
    max_iterations: int                 # 最大迭代次数
    message_id: str                     # 消息 ID
    done: bool                          # 是否完成

# 节点
init_node               # 初始化 Agent 引擎 + 构建 System Prompt
think_node              # LLM 思考（决定是否需要工具调用）
tool_call_node          # 执行工具调用
observe_node            # 观察工具结果
final_answer_node       # 生成最终回答

# 工具集
knowledge_search        # 知识库检索（复用 RetrievalStrategy）
final_answer            # 结束工具 — LLM 通过 tool_call 返回最终答案

# final_answer 工具机制：
# - LLM 在 think 阶段决定调用 final_answer(answer="...")
# - tool_call_node 检测到 final_answer → 提取 answer 字段 → 设置 done=True
# - 路由条件判断 done=True → 跳出循环 → END
# - 如果 LLM 既不调用工具也不调用 final_answer → 达到 max_iterations 后强制结束

# 路由条件函数
def should_continue(state: AgentChatState) -> str:
    """判断是否继续工具调用循环"""
    # 1. 检查是否达到最大迭代次数
    if state["iteration"] >= state["max_iterations"]:
        return "end"
    # 2. 检查最后一条消息是否有 tool_calls
    last_message = state["messages"][-1]
    if hasattr(last_message, "tool_calls") and last_message.tool_calls:
        # 检查是否是 final_answer
        for tc in last_message.tool_calls:
            if tc["name"] == "final_answer":
                return "end"
        return "continue"
    # 3. LLM 没有调用工具 → 视为最终回答
    return "end"

# 图结构
START → init → think → should_continue
                        ├─ "continue" → tool_call → observe → think
                        └─ "end" → END
```

# backend/app/chat/graphs/quick_chat_graph.py

async def retrieve_node(state: QuickChatState) -> dict:
    """向量检索，支持多 KB 串行合并"""
    all_hits = []
    for kb_id in state["knowledge_base_ids"]:
        try:
            hits = await retrieval_strategy.retrieve(
                query=state["normalized_query"],
                knowledge_base_id=kb_id,
                top_k=5
            )
            all_hits.extend(hits)
        except Exception as e:
            # 单 KB 失败不阻断
            emit_error(kb_id=kb_id, code="RETRIEVE_FAILED", message=str(e))

    # 按 score 降序排序，截断 top N
    all_hits.sort(key=lambda h: h.score, reverse=True)
    top_hits = all_hits[:10]

    return {
        "retrieval_attempts": [RetrievalAttempt(
            query=state["normalized_query"],
            hits=top_hits,
            attempt=1
        )],
    }
```

async def generate_answer_node(state: QuickChatState) -> dict:
    """LLM 流式生成回答"""
    llm = get_llm_adapter(session=config["configurable"]["db_session"], user_id=config["configurable"]["user_id"])

    system_prompt = """你是一个知识库问答助手。基于以下检索到的上下文回答用户问题。
如果上下文中没有相关信息，请明确说明。回答时请引用来源。

上下文：
{context}""".format(context=state["context"])

    messages = [
        SystemMessage(content=system_prompt),
        HumanMessage(content=state["query"])
    ]

    # 流式调用
    answer_parts = []
    async for chunk in llm.astream(messages):
        answer_parts.append(chunk.content)
        emit_text_delta(message_id=state["message_id"], delta=chunk.content)

    return {"answer": "".join(answer_parts)}
```

#### 5.4.3 工具实现（新增）

```python
# backend/app/chat/graphs/tools.py

async def knowledge_search_tool(query: str, knowledge_base_ids: list[str], config: RunnableConfig) -> str:
    """知识库检索工具 — Agent 模式调用"""
    session = config["configurable"]["db_session"]
    retrieval_strategy = NaiveV1RetrievalStrategy(session)

    all_hits = []
    for kb_id in knowledge_base_ids:
        try:
            hits = await retrieval_strategy.retrieve(query=query, knowledge_base_id=kb_id, top_k=5)
            all_hits.extend(hits)
        except Exception:
            continue  # 单 KB 失败不阻断

    if not all_hits:
        return "未找到相关信息"

    # 格式化为文本返回给 LLM
    all_hits.sort(key=lambda h: h.score, reverse=True)
    results = []
    for i, hit in enumerate(all_hits[:5], 1):
        results.append(f"[{i}] {hit.content} (来源: {hit.document_name})")
    return "\n\n".join(results)

def final_answer_tool(answer: str) -> str:
    """最终回答工具 — LLM 通过此工具返回答案"""
    return answer

def build_system_prompt(context: str | None = None, mode: str = "quick") -> str:
    """构建 System Prompt"""
    if mode == "quick":
        base = "你是一个知识库问答助手。基于检索到的上下文回答用户问题。如果上下文中没有相关信息，请明确说明。回答时请引用来源。"
        if context:
            return f"{base}\n\n上下文：\n{context}"
        return base
    elif mode == "agent":
        return (
            "你是一个智能知识库助手。你可以通过工具检索知识库来回答用户问题。\n"
            "当你需要查找信息时，使用 knowledge_search 工具。\n"
            "当你已有足够信息回答时，使用 final_answer 工具给出最终答案。\n"
            "每次回答前，先思考是否需要检索更多信息。"
        )
    return "你是一个 helpful assistant。"
```

# backend/app/chat/graphs/agent_chat_graph.py

async def think_node(state: AgentChatState, config: RunnableConfig) -> dict:
    """Agent 思考阶段 — 决定是否需要工具调用"""
    llm = get_llm_adapter(
        session=config["configurable"]["db_session"],
        user_id=config["configurable"]["user_id"],
    )

    tools = [
        Tool(
            name="knowledge_search",
            description="从知识库中检索相关信息",
            func=knowledge_search_tool,
        ),
        Tool(
            name="final_answer",
            description="给出最终答案并结束对话",
            func=final_answer_tool,

### 5.7 流式管道全链路（queue → facade → service → SSE）

现有 `chat_service.py` 的 `stream()` 使用 `_iter_tokens`（按空格分割伪流式），需重构为真正的流式管道。

**完整数据流：**
```
LangGraph astream_events
  → StreamEventCallback 推入 asyncio.Queue
  → ChatFacade.stream() 从 queue 读取并 yield
  → ChatService.stream() 透传 yield
  → API Route SSE 层 yield f"data: {json.dumps(event)}\n\n"
  → BFF parseSse() → mapBackendEventToUiMessageChunkSse()
  → AI SDK useChat() 渲染
```

```python
# backend/app/chat/facade.py — ChatFacade.stream()

async def stream(self, query, knowledge_base_ids, mode, message_id) -> AsyncIterator[dict]:
    """从 asyncio.Queue 读取 StreamEvent 并 yield"""
    event_queue = asyncio.Queue()

    # 后台任务运行 LangGraph 图
    graph_task = asyncio.create_task(
        self._run_graph(query, knowledge_base_ids, mode, message_id, event_queue)
    )

    # 从队列读取事件，直到图运行完成
    while True:
        try:
            event = await asyncio.wait_for(event_queue.get(), timeout=60)
        except asyncio.TimeoutError:
            yield {"type": "error", "message": "响应超时"}
            break

        yield event

        if event.get("type") == "done":
            break

    # 确保后台任务完成（捕获异常）
    try:
        await graph_task
    except Exception as e:
        yield {"type": "error", "message": str(e)}

async def _run_graph(self, query, kb_ids, mode, message_id, event_queue):
    """运行 LangGraph 图，所有事件推入 queue"""
    state = self._build_initial_state(query, kb_ids, mode, message_id)

    await event_queue.put({"type": "start", "message_id": message_id})

    async with get_db_session() as session:
        callback = StreamEventCallback(event_queue, message_id)
        config = RunnableConfig(configurable={
            "stream_callback": callback,
            "db_session": session,
        })

        graph = self._get_graph(mode)
        async for event in graph.astream_events(state, config=config, version="v2"):
            if event["event"] == "on_tool_start":
                await event_queue.put({
                    "type": "tool-call",
                    "tool_name": event["name"],
                    "args": event["data"].get("input"),
                })
            elif event["event"] == "on_tool_end":
                await event_queue.put({
                    "type": "tool-result",
                    "tool_name": event["name"],
                    "output": str(event["data"].get("output")),
                })

    await event_queue.put({"type": "done", "message_id": message_id})

# backend/app/services/chat_service.py — 重构后

async def stream(self, request: ChatRequest) -> AsyncIterator[dict]:
    """透传 facade 的流式事件"""
    facade = self._build_facade(request)

    async for event in facade.stream(
        query=request.query,
        knowledge_base_ids=request.knowledge_base_ids,
        mode=request.mode,
        message_id=request.message_id,
    ):
        yield event
```

**关键设计：**
- `asyncio.Queue` 是 LangGraph 后台任务与 SSE 前台消费者之间的桥梁
- `asyncio.create_task` 运行图，避免阻塞 SSE 响应
- `wait_for(queue.get(), 60)` 防止消费者永久挂起
- StreamEvent 类型：`start`, `text-delta`, `citation`, `step`, `tool-call`, `tool-result`, `error`, `done`

### 5.8 流式事件发射机制（LangGraph 内）

LangGraph 节点是纯函数（输入 state → 输出 state），流式输出需要副作用机制。

**方案：使用 `RunnableConfig` 回调 + `astream_events`**

class StreamEventCallback(AsyncCallbackHandler):
    """将 LLM 流式 token 转为 StreamEvent，推入队列"""

    def __init__(self, event_queue: asyncio.Queue, message_id: str):
        self.queue = event_queue
        self.message_id = message_id

    async def on_llm_new_token(self, token: str, **kwargs):
        await self.queue.put({
            "type": "text-delta",
            "message_id": self.message_id,
            "delta": token,

    messages = [SystemMessage(content=system_prompt), HumanMessage(content=state["query"])]

    answer_parts = []
    async for chunk in llm.astream(messages):
        answer_parts.append(chunk.content)
        await callback.on_llm_new_token(chunk.content)

            pass
        elif event["event"] == "on_tool_start":
            await event_queue.put({"type": "tool-call", ...})

            await event_queue.put({"type": "tool-result", ...})

- 节点是纯函数，直接 emit 是副作用，违反 LangGraph 设计原则
- `astream_events` 是 LangGraph 官方推荐的流式观测接口
- 回调机制解耦了"生成逻辑"和"流式传输逻辑"

### 5.10 上下文窗口管理

Agent 模式下 messages 随迭代增长，需控制 token 数。

```python
# backend/app/chat/graphs/agent_chat_graph.py

import tiktoken

MAX_CONTEXT_TOKENS = 4000  # 预留给 LLM 输出

# provider_type → tiktoken encoding 映射
# 非 OpenAI 模型使用 cl100_base 作为近似（误差 ±10%，可接受）
ENCODING_MAP = {
    "openai": None,      # 由 model_id 动态查找
    "anthropic": "cl100_base",
    "dashscope": "cl100_base",
}

def count_tokens(messages: list[BaseMessage], provider_type: str = "openai", model_id: str = "gpt-4") -> int:
    """计算消息列表的 token 数

    OpenAI 模型使用精确 encoding，其他模型使用 cl100_base 近似。
    误差在 ±10% 内，对滑动窗口裁剪足够。
    """
    enc_name = ENCODING_MAP.get(provider_type, "cl100_base")
    if enc_name is None:
        # OpenAI: 按 model_id 查找精确 encoding
        try:
            encoding = tiktoken.encoding_for_model(model_id)
        except KeyError:
            encoding = tiktoken.get_encoding("cl100_base")
    else:
        encoding = tiktoken.get_encoding(enc_name)

    total = 0
    for msg in messages:
        total += len(encoding.encode(msg.content))
    return total

def trim_messages(messages: list[BaseMessage], max_tokens: int = MAX_CONTEXT_TOKENS,
                  provider_type: str = "openai", model_id: str = "gpt-4") -> list[BaseMessage]:
    """滑动窗口：保留 system prompt + 最近的消息，丢弃中间的"""
    if count_tokens(messages, provider_type, model_id) <= max_tokens:
        return messages

    # 保留第一条（system prompt）和最后 N 条
    system_msg = messages[0]
    recent = messages[1:]

    # 从最近的消息开始保留，直到 token 超限
    kept = []
    total = count_tokens([system_msg])
    for msg in reversed(recent):
        msg_tokens = count_tokens([msg])
        if total + msg_tokens > max_tokens:
            break
        kept.insert(0, msg)
        total += msg_tokens

    return [system_msg] + kept

# 在 think_node 调用前裁剪
async def think_node(state: AgentChatState, config: RunnableConfig) -> dict:
    session = config["configurable"]["db_session"]
    user_id = config["configurable"]["user_id"]
    llm = get_llm_adapter(session=session, user_id=user_id)

    # 裁剪消息列表，避免超出上下文窗口
    messages = trim_messages(
        state["messages"],
        provider_type=llm.provider_type,
        model_id=llm.model_id,
    )

    llm_with_tools = llm.bind_tools(tools)
    response = await llm_with_tools.ainvoke(messages)

    return {"messages": [response], "iteration": state["iteration"] + 1}
```

### 5.11 LLM 超时重试策略

**注意：`@retry` 不能直接用于 `async generator`（astream）。** tenacity 重试会重新调用函数，但调用方可能已消费了部分 token，导致重复输出。解决方案：`astream` 内部捕获超时并 yield 错误，`ainvoke` 使用 tenacity 重试。

```python
# backend/app/chat/infrastructure/llm_adapter.py

import asyncio
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type

LLM_TIMEOUT_SECONDS = 30
LLM_MAX_RETRIES = 1

class LLMAdapter:
    """统一 LLM 调用接口"""

    async def astream(self, messages: list[BaseMessage]) -> AsyncIterator[str]:
        """流式调用 — 不重试（避免重复输出已消费的 token），超时直接报错"""
        llm = self._build_llm()

        try:
            async with asyncio.timeout(LLM_TIMEOUT_SECONDS):
                async for chunk in llm.astream(messages):
                    yield chunk.content
        except asyncio.TimeoutError:
            yield f"\n\n[LLM 响应超时（{LLM_TIMEOUT_SECONDS}s），请重试]"
        except Exception as e:
            yield f"\n\n[LLM 调用失败: {str(e)}]"

    @retry(
        stop=stop_after_attempt(LLM_MAX_RETRIES + 1),  # 首次 + 1次重试
        wait=wait_exponential(multiplier=1, min=1, max=10),
        retry=retry_if_exception_type((asyncio.TimeoutError, ConnectionError)),
    )
    async def ainvoke(self, messages: list[BaseMessage]) -> str:
        """非流式调用 — 带重试（无中间状态，安全重试）"""
        llm = self._build_llm()

        try:
            async with asyncio.timeout(LLM_TIMEOUT_SECONDS):
                response = await llm.ainvoke(messages)
                return response.content
        except asyncio.TimeoutError:
            raise  # tenacity 重试
        except Exception as e:
            return f"[LLM 调用失败: {str(e)}]"
```

**降级策略：**
- **流式（astream）：** 超时/错误 → 直接 yield 错误消息，不重试（避免重复输出）
- **非流式（ainvoke）：** 超时 → 重试 1 次（指数退避 1s）→ 仍失败 → 返回错误
- 非超时错误（如 API key 无效）→ 不重试，直接报错

### 5.12 LLM Provider 选择逻辑

**数据模型关系：**

```python
# backend/app/chat/infrastructure/llm_adapter.py

async def get_llm_adapter(session, user_id: str) -> LLMAdapter:
    """从 models + providers 表获取 LLM 配置

    选择逻辑：
    1. 查找该用户下第一个 is_active=True 的 LLM 类型 Model
    2. 通过 Model.provider_id 关联到 Provider（获取 api_key、base_url）
    3. 如果没有 → 报错提示用户配置 LLM provider

    注意：复用调用方传入的 session，不自行创建新 session（避免连接池耗尽）
    """
    from app.models.model import Model
    from app.models.provider import Provider
    from sqlalchemy import select
        # 查找用户第一个活跃的 LLM model
        stmt = (
            select(Model)
            .join(Provider, Model.provider_id == Provider.id)
            .where(
                Model.model_type == "llm",
                Provider.user_id == user_id,
                Provider.is_active == True,
                Provider.deleted_at.is_(None),
            )
            .limit(1)
        )
        result = await session.execute(stmt)
        model = result.scalar_one_or_none()

        if not model:
            raise ValueError("未配置 LLM Model，请在设置中添加 LLM 类型的模型和 Provider")

        # 获取关联的 Provider
        provider = await session.get(Provider, model.provider_id)

        return LLMAdapter(
            provider_type=provider.provider_type,   # openai / anthropic / dashscope
            model_id=model.model_id,                # gpt-4 / deepseek-chat / ...
            api_key_encrypted=provider.api_key_encrypted,
            base_url=provider.base_url,
            context_window=model.context_window,     # 用于上下文窗口管理
        )
```

### 5.14 模块化检索抽象（Retriever）

**现状：** `RetrievalStrategy` 只用于读操作（`KnowledgeBaseService.retrieve()`），写操作不依赖它。可以安全替换。

```python
# backend/app/retrieval/vector_retriever.py

class VectorRetriever:
    """向量检索 — 包装现有 NaiveV1RetrievalStrategy"""
    retriever_id = "naive.vector"
    source_type = "vector"

    def __init__(self, db_session, embedding_client, connector_factory):
        self._session = db_session
        self._embed = embedding_client
        self._connectors = connector_factory

    async def retrieve(self, *, query: str, config: RetrievalConfig) -> RetrievalResult:
        start = time.monotonic()

        # 1. embed query
        query_vector = (await self._embed.embed_texts([query]))[0]

        # 2. 获取 connector 和 collection
        connector, collection = await self._get_connector(config.knowledge_base_id)

        # 3. 向量搜索
        hits = await connector.search(
            collection=collection,
            knowledge_base_id=config.knowledge_base_id,
            query_vector=query_vector,
            top_k=config.top_k,
            score_threshold=config.score_threshold,
        )

        return RetrievalResult(
            hits=hits,
            source="vector",
            latency_ms=int((time.monotonic() - start) * 1000),
        )
```

- `NaiveV1RetrievalStrategy` 保留但标记 `@deprecated`，下个迭代清理

class ToolRegistry:
    def __init__(self):
        self._tools: dict[str, ToolDefinition] = {}
        self._handlers: dict[str, ToolHandler] = {}
        self._providers: list[ToolProvider] = []

    async def register_provider(self, provider: ToolProvider):
        """从 provider 批量注册工具"""
        self._providers.append(provider)
        for tool_def in await provider.list_tools():
            self._tools[tool_def.name] = tool_def
            # handler 通过 provider.invoke 路由
            self._handlers[tool_def.name] = lambda args, p=provider, n=tool_def.name: p.invoke(n, args)

    def set_mode_allowlist(self, mode: str, tool_names: set[str]):
        self._mode_allowlists[mode] = tool_names

    def to_langchain_tools(self) -> list:
        """转换为 LangChain Tool 列表，供 Agent 使用"""
        from langchain_core.tools import Tool
        tools = []
        for name, tool_def in self._tools.items():
            tools.append(Tool(
                name=name,
                description=tool_def.description,
                func=self._handlers[name],
                # args_schema 可从 tool_def.parameters (JSON Schema) 生成
            ))
        return tools

### 5.17 LangFlow / 外部工作流对接

**用户需求：** LangFlow 配置的工作流作为 MCP 或 API 源接入到 RAG flow 中。

**方案：LangFlow 作为 ToolProvider 接入**

LangFlow 工作流暴露 HTTP API，BuildTest AI 通过 `LangFlowToolProvider` 调用：

```python
# backend/app/tools/langflow_provider.py

class LangFlowToolProvider:
    """LangFlow 工作流提供者 — 将 LangFlow DAG 包装为工具"""
    provider_id = "langflow:default"
    source_type = "langflow"

    def __init__(self, base_url: str, api_key: str):
        self.base_url = base_url.rstrip("/")
        self.api_key = api_key
        self._tools_cache: list[ToolDefinition] | None = None

    async def list_tools(self) -> list[ToolDefinition]:
        """从 LangFlow API 获取可用工作流列表"""
        if self._tools_cache:
            return self._tools_cache

        async with httpx.AsyncClient() as client:
            resp = await client.get(
                f"{self.base_url}/api/v1/flows",
                headers={"Authorization": f"Bearer {self.api_key}"},
            )
            flows = resp.json()

        self._tools_cache = [
            ToolDefinition(
                name=f"langflow:{flow['id']}",
                description=flow.get("description", f"LangFlow workflow: {flow['name']}"),
                parameters=flow.get("input_schema", {"type": "object", "properties": {
                    "input": {"type": "string", "description": "输入文本"},
                }}),
                category="api",
                source_type="langflow",
            )
            for flow in flows
        ]
        return self._tools_cache

    async def invoke(self, tool_name: str, arguments: dict) -> dict:
        """调用 LangFlow 工作流"""
        flow_id = tool_name.removeprefix("langflow:")

        async with httpx.AsyncClient(timeout=60) as client:
            resp = await client.post(
                f"{self.base_url}/api/v1/run/{flow_id}",
                headers={"Authorization": f"Bearer {self.api_key}"},
                json={"input_value": arguments.get("input", ""), "input_type": "chat"},
            )
            resp.raise_for_status()
            result = resp.json()

        return {
            "output": result.get("outputs", [{}])[0].get("outputs", [{}])[0].get("results", {}).get("message", {}).get("text", ""),
            "raw": result,
        }
```

### 5.19 模型配置架构（按用途绑定）

**现状问题：** `knowledge_bases` 只有 `embedding_model_id` 一个 FK，锁死了"一个 KB 一个模型"。LLM、Rerank、VLM 无配置入口。前端 Provider 页面选模型时只能选同一 Provider 的。

**MVP 决策：**
- 保留 `embedding_model_id`（兼容旧数据），新增 `kb_model_configs` 扩展表
- Agent 按 `agent_id` 粒度配置（`quick_chat` / `smart_agent`），SubAgent 继承主模型
- `model_type` 扩展为 `llm / embedding / rerank / vlm`，rerank 独立类型

#### 5.19.2 运行时模型解析

```python
# backend/app/chat/infrastructure/model_resolver.py

class ModelResolver:
    """运行时模型解析器 — 统一获取模型配置"""

    def __init__(self, db_session, user_id: str):
        self._session = db_session
        self._user_id = user_id

    async def get_kb_model(self, kb_id: uuid.UUID, purpose: str) -> ModelWithProvider:
        """获取知识库指定用途的模型

        优先级：
        1. kb_model_configs(kb_id, purpose) → model → provider
        2. 如果 purpose == "embedding"，降级到 kb.embedding_model_id
        3. 都没有 → ValueError
        """
        # 1. 查配置表
        stmt = select(KbModelConfig).where(
            KbModelConfig.knowledge_base_id == kb_id,
            KbModelConfig.purpose == purpose,
        )
        config = (await self._session.execute(stmt)).scalar_one_or_none()
        if config:
            return await self._resolve_model(config.model_id)

        # 2. embedding 降级
        if purpose == "embedding":
            kb = await self._session.get(KnowledgeBase, kb_id)
            if kb and kb.embedding_model_id:
                return await self._resolve_model(kb.embedding_model_id)

        raise ValueError(f"知识库未配置 {purpose} 模型，请在知识库设置中配置")

    async def get_agent_model(self, agent_id: str) -> ModelWithProvider:
        """获取 Agent 的 LLM 模型

        优先级：
        1. agent_model_configs(user_id, agent_id)
        2. 用户第一个 is_active 的 llm model（兜底）
        """
        stmt = select(AgentModelConfig).where(
            AgentModelConfig.user_id == self._user_id,
            AgentModelConfig.agent_id == agent_id,
        )
        config = (await self._session.execute(stmt)).scalar_one_or_none()
        if config:
            return await self._resolve_model(config.model_id)

        # 兜底
        return await self._get_fallback_llm()

    async def _resolve_model(self, model_id: uuid.UUID) -> ModelWithProvider:
        model = await self._session.get(Model, model_id)
        provider = await self._session.get(Provider, model.provider_id)
        return ModelWithProvider(model=model, provider=provider)
```

#### 5.20.2 Agent 工具调用展示

**流式渲染顺序：**
1. 用户消息立即显示
2. Assistant 消息开始流式渲染（文本 token）
3. 遇到 tool_call 事件 → 插入 tool-call 卡片（状态: 运行中）
4. 遇到 tool_result 事件 → 更新卡片状态为完成，插入 tool-result 卡片
5. 最终回答继续流式渲染

**tool-call 卡片规格：**
- 宽度：消息区域 100%（与普通消息同宽）
- 内边距：12px
- 圆角：8px
- 背景：`bg-muted`（比消息背景深一级）
- 函数名：等宽字体，14px，`font-medium`
- 参数摘要：12px，`text-muted-foreground`，截断到 80 字符
- 状态指示器：8px 圆点，左侧对齐函数名

**tool-result 卡片规格：**
- 与 tool-call 卡片同级，不额外缩进
- 默认折叠：只显示「返回 N 条结果 (Xs)」+ 展开箭头
- 展开后：格式化 JSON + 语法高亮（等宽字体，key 灰色，string 绿色，number 蓝色），最大高度 300px，可滚动
- 复用前端已有的 markdown 代码块渲染（`AssistantMarkdownMessage`）处理 JSON 展示

**Provider 下拉筛选逻辑：**
- 只显示已配置 API key 的 Provider（`providers.api_key_encrypted IS NOT NULL`）
- 如果用户没有任何已配置的 Provider，下拉为空 + 提示「请先在 Provider 页面配置 API key」
- Model 下拉按选中的 Provider 过滤，只显示该 Provider 下 `is_active=true` 的模型
- Embedding 组只显示 `model_type=embedding` 的模型；LLM 组只显示 `model_type=llm`

## 六、实施计划

> **CEO Review 调整（2026-04-28）：**
> - Phase 顺序调整：LLM 先行，建表后置。先出可见成果，再投资抽象层。
> - Retriever 协议保留（Phase 2），ToolProvider 延后（Agent 用 LangChain Tools）。
> - 每个 Phase 结束有可演示的 checkpoint。

### Phase 1：Quick Chat + LLM 真实生成（~3 天）

> 最高优先级：让用户能从文档中获得 LLM 生成的真实回答。这是整个项目的核心价值验证点。

1. 安装依赖：`langgraph`, `langchain-core`, `langchain-openai`(或对应 provider)，解决依赖冲突（0.5d buffer）
2. **新增 `LLMAdapter`**（见 §5.5）— 复用现有 `providers` + `models` 表，无需新增列。通过 `get_llm_adapter(session, user_id)` 获取用户第一个活跃 LLM 模型（兜底逻辑）
3. **实现 Quick Chat LangGraph 图**（见 §5.2）— `normalize → retrieve → [rewrite_retry] → assemble → generate → finalize`
4. **流式管道重构**（见 §5.7-5.8）— `asyncio.Queue` 桥接 LangGraph 后台任务与 SSE 前台消费者，替代 `_iter_tokens` 伪流式
5. LLM 超时重试策略（见 §5.11）— 流式不重试（避免重复 token），非流式重试 1 次
6. 测试：有上下文生成、无上下文降级、空结果重试、LLM 超时处理

**Checkpoint：** `POST /chat/stream` with `mode=quick` 返回 LLM 生成的流式回答，端到端延迟 < 5s

### Phase 2：Retriever 抽象层（~1.5 天）

> 在 Quick Chat 能跑之后，引入检索抽象层，为 GraphRetriever 做准备。

1. **引入 `Retriever` 协议 + 注册表**（见 §5.14），实现 `VectorRetriever`（包装现有 `NaiveV1RetrievalStrategy`）
2. **引入 `AgeGraphRetriever` 桩**（见 §5.16），接口定义 + 空实现（返回空结果，不阻断流程）
3. 重构 Quick Chat 节点：从直接调用 `NaiveV1RetrievalStrategy` 改为通过 `RetrieverRegistry` 获取检索器
4. 测试：向量检索走新 Retriever 接口，行为与 Phase 1 一致

**Checkpoint：** 检索走统一 Retriever 接口，GraphRetriever 桩注册成功但返回空结果

### Phase 3：Agent Chat（~3.5 天）

> Agent 多步推理能力。使用 LangChain Tools 直接调用，不引入 ToolProvider 抽象层。

1. 实现 Agent Chat LangGraph 图（见 §5.2）— `init → think → tool_call → observe` 循环
2. 工具使用 LangChain `Tool` 对象直接定义（`knowledge_search` + `final_answer`），不走 ToolProvider 协议
3. Agent LLM 使用 `get_llm_adapter(session, user_id)` 兜底逻辑（同 Phase 1）
4. 实现对话历史管理（见 §5.9）— 前端 POST-back 模式，历史最大 20 条
5. 实现上下文窗口管理（见 §5.10）— tiktoken 计数 + 滑动窗口裁剪
6. 更新 `chat_service.py` 的 mode 分发，`mode=agent` 走新图
7. Agent 最大迭代次数硬编码 3 次（MVP），后续可配置
8. 测试：单步推理、多步推理、工具调用循环、达到上限降级

**Checkpoint：** `POST /chat/stream` with `mode=agent` 返回多步推理结果，工具调用正确

### Phase 4：模型配置系统（~2 天）

> 从"用兜底 LLM"升级为"按用途/按 Agent 精确配置模型"。

## 七、数据流与错误处理

### 7.1 Quick Chat 数据流

```
Happy Path:
  用户查询 → normalize → retrieve(多KB) → 有结果 → assemble → LLM生成 → 流式输出 → done

Nil Path (查询为空):
  用户查询(空) → normalize(空) → 返回错误 "查询不能为空"

Empty Path (检索无结果):
  用户查询 → normalize → retrieve(无结果) → rewrite → retrieve(仍无结果) → 降级回答 "未找到相关信息"

Error Path (LLM 调用失败):
  用户查询 → ... → LLM生成(超时) → 重试1次(指数退避1s) → 仍失败 → emit error 事件 → 返回错误提示 "LLM 服务暂时不可用，请稍后重试"
```

### 7.2 Agent Chat 数据流

```
Happy Path:
  用户查询 → init → think → tool_call(knowledge_search) → observe → think → final_answer → done

Multi-step Path:
  用户查询 → init → think → tool_call → observe → think → tool_call → observe → think → final_answer

Max Iteration Path:
  用户查询 → init → [think → tool_call → observe]×N → 达到上限 → 强制 final_answer

Tool Error Path:
  用户查询 → init → think → tool_call(失败) → observe(错误) → think(重试或降级) → final_answer
```

### 8.1 安全

- LLM API Key 通过 `providers` 表加密存储，运行时解密
