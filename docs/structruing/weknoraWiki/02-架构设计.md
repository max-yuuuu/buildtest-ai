# 架构设计

## 整体架构

WeKnora 采用 **分层架构** 设计，遵循 **领域驱动设计 (DDD)** 和 **依赖倒置** 原则。

## 核心架构模式

### 1. 依赖注入容器

项目使用 **Uber Dig** 作为依赖注入容器，位于 [`internal/container/`](../internal/container/container.go)。

```go
// internal/container/container.go
func BuildContainer(c *dig.Container) *dig.Container {
    // 注册所有依赖
    c.Provide(config.NewConfig)
    c.Provide(logger.NewLogger)
    c.Provide(database.NewDB)
    c.Provide(service.NewKnowledgeService)
    // ...
    return c
}
```

### 2. Repository 模式

所有数据访问通过 Repository 层抽象，支持多种数据库实现。

## 核心数据流

### 问答流程 (RAG 模式)

```
用户提问
   │
   ▼
┌──────────────────────────────────────────────────────┐
│  1. 查询理解 (Query Understanding)                  │
│     - 意图识别                                      │
│     - 实体提取                                      │
│     - 查询改写                                      │
└───────────────────────┬──────────────────────────────┘
                        │
                        ▼
┌──────────────────────────────────────────────────────┐
│  2. 知识检索 (Knowledge Retrieval)                  │
│     - 向量检索                                      │
│     - 关键词检索 (BM25)                             │
│     - 混合检索 (Hybrid)                             │
│     - 知识图谱增强 (GraphRAG)                       │
└───────────────────────┬──────────────────────────────┘
                        │
                        ▼
┌──────────────────────────────────────────────────────┐
│  3. 重排序 (Reranking)                              │
│     - 相关性评分                                    │
│     - 去重过滤                                      │
└───────────────────────┬──────────────────────────────┘
                        │
                        ▼
┌──────────────────────────────────────────────────────┐
│  4. 答案生成 (Answer Generation)                    │
│     - 上下文构建                                    │
│     - LLM 调用                                      │
│     - 流式输出                                      │
└───────────────────────┬──────────────────────────────┘
                        │
                        ▼
                      响应用户
```

## 核心服务架构

                ▼

### 2. 聊天管道 (Chat Pipeline)

// 流水线步骤

## 技术选型原则

1. **模块化**: 每个组件可以独立替换
2. **可扩展**: 支持插件式扩展
3. **高性能**: 异步处理 + 缓存优化
4. **安全**: 数据加密 + 权限控制
5. **易部署**: Docker + K8s 支持
