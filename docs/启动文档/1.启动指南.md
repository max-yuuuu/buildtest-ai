## 方式一：Docker Compose 启动（推荐）

### 前置条件

- [Docker](https://docs.docker.com/get-docker/) 已安装
- [Docker Compose](https://docs.docker.com/compose/install/) 已安装

### 启动步骤

```bash
# 1. 进入项目根目录
cd /Users/xinjie03/Desktop/xinjie/buildtest-ai

# 2. 复制环境变量模板
cp .env.example .env
cp env/common.env.example env/common.env
cp env/dev.docker.env.example env/dev.docker.env
cp env/prod.env.example env/prod.env

# 3. 编辑 .env 填写必要配置（详见下方环境变量配置说明）
vim .env

# 4. 启动所有服务（compose.base + compose.dev）
make dev

# 5. 查看服务状态
docker compose -f compose.base.yml -f compose.dev.yml ps
```

### 停止服务

```bash
docker compose -f compose.base.yml -f compose.dev.yml down
```

### 重建并启动后端

```bash
docker compose -f compose.base.yml -f compose.dev.yml build backend
docker compose -f compose.base.yml -f compose.dev.yml up -d backend
```

## 方式二：本地前后端分别启动

### 前置条件

- uv（Python 版本管理器）
- Python 3.12（建议固定 3.12，【3.13 出现依赖编译问题】）
- Node.js 18+
- pnpm
- PostgreSQL 16（需启用 pgvector 扩展）
- Redis 7+
- Qdrant 向量数据库

### 2.1 后端启动

```bash
# 进入后端目录
cd /Users/xinjie03/Desktop/xinjie/buildtest-ai/backend

# 使用 uv 安装 Python 3.12（如尚未安装）
uv python install 3.12

# 创建虚拟环境（指定 Python 3.12）
uv venv .venv --python 3.12

# 激活虚拟环境
source .venv/bin/activate

# 安装依赖
uv pip install -r requirements.txt

# 配置环境变量
cp ../.env.example .env
# 编辑 .env 填写必要配置（详见下方环境变量配置说明）

# 运行数据库迁移
alembic upgrade head

# 启动开发服务器
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

### 2.2 Celery Worker 启动（本地模式必需）

处理文档解析、向量入库等异步任务。本地跑后端时，请与后端使用同一套环境变量（尤其 `DATABASE_URL`、`REDIS_URL`、`UPLOAD_DIR`）。

```bash
# 在后端目录下
celery -A app.core.celery_app worker -l info
```

### 2.3 前端启动

```bash
# 进入前端目录
cd /Users/xinjie03/Desktop/xinjie/buildtest-ai/frontend

# 安装依赖
pnpm install

# 配置环境变量
# 前端使用 NextAuth，推荐直接复制 frontend/.env（或使用 .env.local）
cp ../.env.example .env.local
# 或：cp .env .env.local（若你已在 frontend/.env 维护本地配置）

# 启动开发服务器
pnpm dev
```

### 2.4 本地服务依赖启动

若不使用 Docker，需手动启动 PostgreSQL、Redis、Qdrant：

```bash
# PostgreSQL（需安装 pgvector 扩展）
# 创建数据库和用户
createuser buildtest
createdb -O buildtest buildtest
# 在数据库中启用 pgvector
psql -U buildtest -d buildtest -c "CREATE EXTENSION IF NOT EXISTS vector;"

# Redis
redis-server

# Qdrant
docker run -p 6333:6333 -p 6334:6334 qdrant/qdrant:v1.11.0
```

## 方式三：混合启动（常用调试模式）

适用于“只改前后端代码，但依赖服务继续用 Docker”的场景。

### 常见组合

1. **本地前端 + Docker 后端与依赖**
2. **本地后端 + Docker 前端与依赖**
3. **本地前端 + 本地后端 + Docker 仅依赖（Postgres/Redis/Qdrant）**

### 推荐命令（组合 3）

```bash
# 项目根目录（推荐）：一条命令启动 infra + backend(host) + worker(host) + frontend(host)
make up
```

也可以分别启动：

```bash
make infra
make backend
make worker
make frontend
```

## Compose 分层矩阵（新）

推荐统一使用以下四种模式：

```bash
# 1) 开发全容器（base + dev）
docker compose -f compose.base.yml -f compose.dev.yml up --build

# 2) 生产编排（base + prod）
docker compose -f compose.base.yml -f compose.prod.yml up -d --build

# 3) 仅基础设施（infra）
docker compose -f compose.infra.yml up -d

# 4) 应用本地 + 基础设施容器
make infra
make backend
make worker
make frontend
```

每次修改 compose 相关文件后，建议先执行：

```bash
make validate-compose
```

发布前检查与回滚建议：

```bash
# 发布前
make validate-compose
make doctor

# 统一入口（推荐）
make dev
```

## 环境变量配置说明

复制 `.env.example` 为 `.env` 后，需填写以下配置项：

### 必需配置

| 变量名 | 说明 | 生成方式 |
|--------|------|----------|
| `NEXTAUTH_SECRET` | NextAuth 会话密钥 | `openssl rand -base64 32` |
| `NEXTAUTH_URL` | NextAuth 访问地址 | 本地开发填 `http://localhost:3000` |
| `APP_ENCRYPTION_KEY` | 应用加密主密钥 | `python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"` |
| `GITHUB_ID` | GitHub OAuth Client ID | [GitHub OAuth App 注册](https://github.com/settings/developers) |
| `GITHUB_SECRET` | GitHub OAuth Client Secret | 同上 |
| `GOOGLE_ID` | Google OAuth Client ID | [Google Cloud Console](https://console.cloud.google.com/) |
| `GOOGLE_SECRET` | Google OAuth Client Secret | 同上 |

### 本地开发默认配置（无需修改）

```bash
DATABASE_URL=postgresql+asyncpg://buildtest:buildtest@localhost:5432/buildtest
REDIS_URL=redis://localhost:6379/0
QDRANT_URL=http://localhost:6333
LOG_LEVEL=INFO
ENVIRONMENT=development
```

补充说明：

- 上述默认值是“**本地进程访问本机服务**”的写法。
- 若服务运行在 Docker 内，容器间通信需使用服务名（如 `postgres`、`redis`、`qdrant`、`backend`）。
- `APP_ENCRYPTION_KEY` 必须固定；更换后历史密文将无法解密。
- Provider API Key（如 OpenAI）在应用内 Provider 管理页配置，不需要写入根目录 `.env`。
- `UPLOAD_DIR` 建议通过 `env/common.env` 配置（默认 `./uploads`），启动脚本会自动转换为仓库根绝对路径，避免 backend/worker 路径不一致。

## 不同启动方式的环境变量对照

下面是最容易出错的几个变量：

| 场景 | `DATABASE_URL` | `REDIS_URL` | `QDRANT_URL` | `BACKEND_URL`（前端 BFF） |
|------|----------------|-------------|--------------|------------------------|
| 全 Docker（前后端都在容器内） | `...@postgres:5432/...` | `redis://redis:6379/0` | `http://qdrant:6333` | `http://backend:8000` |
| 本地前后端 + 本地依赖 | `...@localhost:5432/...` | `redis://localhost:6379/0` | `http://localhost:6333` | `http://localhost:8000` |
| 本地前后端 + Docker 依赖（端口映射到本机） | `...@localhost:5432/...` | `redis://localhost:6379/0` | `http://localhost:6333` | `http://localhost:8000` |

> 原则：**进程在哪儿运行，就从那个进程视角填写地址**。  
> 本地进程看容器服务应走 `localhost:映射端口`；容器进程看同编排服务应走 `service name`。
>
> 当前仓库的真实加载顺序为：
> - host 模式：`env/common.env` → 根 `.env` → `env/dev.host.env`
> - Docker dev 模式：根 `.env` + `env/common.env` + `env/dev.docker.env`
>
> Next.js BFF（`/api/backend/*` 与 `/api/chat`）优先使用显式 `BACKEND_URL`；只有不可达时才回退探测 `localhost` / `backend` / `host.docker.internal`。

### OAuth 回调地址配置

- GitHub OAuth App：**Authorization callback URL** → `http://localhost:3000/api/auth/callback/github`
- Google OAuth：**Authorized redirect URI** → `http://localhost:3000/api/auth/callback/google`

## Chat 验证步骤

下面的验证步骤专门用于本次 Quick Chat 多知识库 + `ai@6` UI stream 链路。

### A. host 模式验证

适用场景：

- `scripts/dev up`
- 或者分别执行 `scripts/dev backend`、`scripts/dev worker`、`scripts/dev frontend`

前置确认：

```bash
# 1) 检查 host 模式 env
scripts/dev print-env

# 2) 关键值应满足
# BACKEND_URL=http://localhost:8000
# DATABASE_URL/REDIS_URL/QDRANT_URL 均指向 localhost
```

验证步骤：

```bash
# 1) 前端与后端健康检查
curl http://localhost:3000
curl http://localhost:8000/healthz

# 2) 浏览器打开 chat 页面
open http://localhost:3000/chat
```

手工验证清单：

1. 顶部能选择“默认主知识库”
2. 输入框中输入 `@知识库名` 后，出现 suggestion / pills
3. 发送后，Network 中请求命中 `POST /api/chat`
4. 请求体为 `knowledge_base_ids[]`，不是旧的 `knowledge_base_id`
5. assistant 回复能持续流式出现，而不是一次性整段返回
6. 若后端返回 citation / step / tool 事件，页面可见对应块

### B. Docker dev 模式验证

适用场景：

- `make dev`
- 或者 `docker compose -f compose.base.yml -f compose.dev.yml up --build`

前置确认：

```bash
# 关键文件
cat env/dev.docker.env

# 关键值应满足
# BACKEND_URL=http://backend:8000
```

验证步骤：

```bash
# 1) 检查 compose 服务
docker compose -f compose.base.yml -f compose.dev.yml ps

# 2) 健康检查（从宿主机）
curl http://localhost:3000
curl http://localhost:8000/healthz
```

手工验证清单与 host 模式相同，但需要额外确认：

1. 前端容器日志中没有持续的 backend 探测失败
2. `/api/chat` 最终命中的 backend 地址是 compose 网络里的 `http://backend:8000`
3. 同一条问题在 Docker dev 与 host 模式下都能得到流式回复

### C. 出问题时优先看哪里

若 chat 页面能开，但发送失败，优先按这个顺序排查：

1. `scripts/dev print-env` 或 `env/dev.docker.env` 中的 `BACKEND_URL`
2. `curl http://localhost:8000/healthz`
3. 浏览器 Network 中 `/api/chat` 的响应头是否包含 `x-vercel-ai-ui-message-stream: v1`
4. 前端服务端日志 / 后端日志中是否出现 `knowledge_base_ids` 或 SSE 事件映射报错

## 常见问题

### 1. 依赖安装失败（psycopg2、asyncpg、tiktoken 等）

检查 Python 版本，确保使用 **Python 3.12**（项目指定的版本）：

```bash
# 查看当前 Python 版本
python --version

# 如使用 Python 3.13，部分包没有预编译 wheel，需从源码编译
# 推荐使用 uv 管理 Python 版本：
uv python install 3.12
uv venv .venv --python 3.12
source .venv/bin/activate
uv pip install -r requirements.txt
```

### 2. 后端启动提示 `.env not found`

确保 `.env` 文件存在于项目根目录：

```bash
cp .env.example .env
```

### 3. 数据库迁移失败

```bash
# 查看当前迁移状态
alembic current

# 手动运行迁移
alembic upgrade head

# 回滚迁移
alembic downgrade -1
```

### 4. 前端无法连接后端

优先确认根目录 env 分层里的 `BACKEND_URL` 配置正确，而不是单独修改 `frontend/.env`：

```bash
# host 模式（frontend/backend 跑在本机，infra 跑在 docker）
env/dev.host.env
BACKEND_URL=http://localhost:8000

# Docker dev 模式（frontend/backend 都在 compose 网络内）
env/dev.docker.env
BACKEND_URL=http://backend:8000
```

若你确实额外维护了 `frontend/.env.local`，请确保不要覆盖上述分层语义。修改后请重启前端 dev server。

### 5. `/api/backend/providers` 返回 500（高频）

这是本地调试时最常见的问题之一，通常不是前端页面本身，而是后端调用链异常。

优先排查：

1. 直接测后端是否可用：

```bash
curl http://localhost:8000/healthz
curl -H "X-User-Id: debug-user" -H "X-User-Email: debug@example.com" http://localhost:8000/api/v1/providers
```

2. 若后端也 500，重点检查 `backend/.env`：
   - 本地后端请使用 `localhost`，不要写 `postgres` / `redis` / `qdrant`
   - 示例：

```bash
DATABASE_URL=postgresql+asyncpg://buildtest:buildtest@localhost:5432/buildtest
REDIS_URL=redis://localhost:6379/0
QDRANT_URL=http://localhost:6333
```

3. 确认依赖服务端口可达：

```bash
nc -vz localhost 5432
nc -vz localhost 6379
nc -vz localhost 6333
```

4. 修改环境变量后，重启前后端。

### 6. 重新构建镜像

```bash
docker compose -f compose.base.yml -f compose.dev.yml build --no-cache
docker compose -f compose.base.yml -f compose.dev.yml up -d
```

### 7. 查看容器日志

```bash
# 后端日志
docker compose -f compose.base.yml -f compose.dev.yml logs -f backend

# Celery Worker 日志
docker compose -f compose.base.yml -f compose.dev.yml logs -f celery-worker

# 所有服务日志
docker compose -f compose.base.yml -f compose.dev.yml logs -f
```

### 8. 上传后状态一直 queued 或提示“原文文件不存在”

常见根因是本地 worker 与 docker worker 同时消费，或 backend/worker 的 `UPLOAD_DIR` 不一致。

排查顺序：

1. 仅保留一套 worker（本地模式建议停掉 docker worker）：

```bash
docker compose -f compose.base.yml -f compose.dev.yml stop celery-worker
```

2. 确认环境一致：

```bash
make print-env
```

确保 backend 与 worker 看到相同的 `DATABASE_URL`、`REDIS_URL`、`UPLOAD_DIR`。

3. 重启 `make up` 后重试失败文档。
